## Key Insights
- Multi-tenant: ALL tables have `companyId` (except users, sessions, accounts, verification, billingAccounts, companyTemplates)
- Compound indexes on `(company_id, ...)` for every tenant-scoped table
- JSONB for flexible config fields (adapterConfig, runtimeConfig, runnerConfig, usageJson)
- Self-referential FKs: agents.reportsTo -> agents.id, issues.parentId -> issues.id, goals.parentId -> goals.id
- Better Auth manages its own tables (users, sessions, accounts, verification) but we define Drizzle schemas for them
- 9 critical indexes for performance

## Requirements
### Functional
- All 35+ table schemas defined in Drizzle
- Proper FK relationships
- Indexes on hot query paths
- Initial migration generatable via `drizzle-kit generate`

### Non-functional
- Neon PostgreSQL compatible
- PgBouncer-safe (no advisory locks in connection pool mode; use direct connection for migrations)

## Implementation Steps

1. **Create `base-model-columns.ts`**
   - Helper functions for common columns:
     ```typescript
     export const idColumn = () => uuid('id').primaryKey().defaultRandom();
     export const companyIdColumn = () => uuid('company_id').notNull();
     export const timestampColumns = () => ({
       createdAt: timestamp('created_at').defaultNow().notNull(),
       updatedAt: timestamp('updated_at').defaultNow().notNull(),
     });
     ```

2. **Create auth tables** (Better Auth managed)
   - `user-model.ts`: users table (id text PK, name, email unique, emailVerified, plan, createdAt, updatedAt)
   - `session-model.ts`: sessions (id, userId, expiresAt, ipAddress, userAgent, token)
   - `account-model.ts`: accounts (id, userId, accountId, providerId, accessToken, refreshToken, idToken, ...)
   - `verification-model.ts`: verification (id, identifier, value, expiresAt)

3. **Create core business tables**
   - `company-model.ts`: Per blueprint 15 columns. Indexes: `(owner_id)`, `(status)`
   - `user-company-model.ts`: Junction (userId, companyId, role). Unique on `(userId, companyId)`
   - `agent-model.ts`: All columns from blueprint. Indexes: `(company_id, status)`, `(company_id, reports_to)`
   - `issue-model.ts`: All columns. Indexes: `(company_id, status)`, `(company_id, assignee_agent_id, status)`, UNIQUE `(identifier)`
   - `heartbeat-run-model.ts`: Per blueprint. Index: `(company_id, agent_id, started_at)`, `(agent_id, status, started_at)`
   - `heartbeat-run-event-model.ts`: bigserial PK, runId FK, seq, eventType, stream, message, payload. Index: `(run_id, seq)`. **Regular table (no partitioning for V1)**
   <!-- Updated: Validation Session 1 - Defer partitioning to post-MVP -->

4. **Create agent supporting tables**
   - `agent-api-key-model.ts`: hash (unique), agentId, companyId, label, revokedAt
   - `agent-runtime-state-model.ts`: agentId (unique per company), currentRunId, totalInputTokens, totalOutputTokens, totalCostCents, lastHeartbeatAt
   - `agent-task-session-model.ts`: agentId, taskId, sessionData (jsonb), updatedAt
   - `agent-wakeup-request-model.ts`: agentId, companyId, source, status, taskId, commentId, approvalId, coalescedCount, claimedAt, completedAt
   - `agent-config-revision-model.ts`: agentId, revisionNumber, adapterConfig (jsonb), runtimeConfig (jsonb), createdBy, createdAt

5. **Create issue supporting tables**
   - `issue-comment-model.ts`: issueId, body, authorType, authorAgentId, authorUserId, runId
   - `issue-attachment-model.ts`: issueId, assetId, filename
   - `label-model.ts`: companyId, name, color. Unique: `(company_id, name)`
   - `issue-label-model.ts`: issueId, labelId (junction)

6. **Create remaining tables**
   - `goal-model.ts`: companyId, parentId (self-ref), title, description, level, status
   - `project-model.ts`: companyId, goalId, name, description, archivedAt
   - `project-workspace-model.ts`: projectId, repoUrl, repoRef, cwd, branch
   - `approval-model.ts`: companyId, type, status, payload (jsonb), requestedByAgentId, resolvedByUserId, resolvedAt
   - `approval-comment-model.ts`: approvalId, body, authorType, authorId
   - `cost-event-model.ts`: companyId, agentId, runId, provider, model, inputTokens, outputTokens, costCents, computeCostCents
   - `activity-model.ts`: companyId, actorType, actorId, action, entityType, entityId, details (jsonb), runId
   - `company-api-key-model.ts`: companyId, provider, encryptedKey, keyHash, label, isValid, lastValidatedAt, lastUsedAt
   - `company-vm-model.ts`: companyId (unique), machineId, status, region, size, volumeId, lastActiveAt
   - `company-template-model.ts`: id (slug PK), name, description, category, agentConfigs (jsonb), goalTemplate, isPublic
   - `asset-model.ts`: companyId, key (S3 key), filename, contentType, sizeBytes
   - `billing-account-model.ts`: userId, plan, stripeCustomerId, credits
   - `usage-record-model.ts`: companyId, month, tokenCostCents, computeCostCents

7. **Create `schemas/index.ts`** barrel export of all tables

8. **Create `database.module.ts`**
   ```typescript
   @Global()
   @Module({
     providers: [{
       provide: 'DRIZZLE',
       useFactory: async (config: ConfigService) => {
         const pool = new Pool({ connectionString: config.get('DATABASE_URL'), max: 10 });
         return drizzle(pool, { schema });
         // Using node-postgres (pg) driver per validation decision
       },
       inject: [ConfigService],
     }],
     exports: ['DRIZZLE'],
   })
   export class DatabaseModule {}
   ```

9. **Create `drizzle.config.ts`**
   ```typescript
   export default defineConfig({
     schema: './src/infrastructure/persistence/schemas/*.ts',
     out: './src/infrastructure/persistence/migrations',
     dialect: 'postgresql',
     dbCredentials: { url: process.env.DATABASE_URL! },
   });
   ```

10. **Generate initial migration**
    - `pnpm --filter @ai-company/backend drizzle-kit generate`
    - Review generated SQL
    - Add migration script to package.json

## Success Criteria
- All 35+ tables defined with correct types, FKs, indexes
- `drizzle-kit generate` produces valid SQL
- Migration applies cleanly to a fresh PostgreSQL instance
- `turbo typecheck` passes

## Risk Assessment
- **FK ordering:** Drizzle handles circular refs via callback syntax: `.references(() => agents.id)`
- **Better Auth table compatibility:** Match exact column names Better Auth expects (snake_case)
- **Large migration:** Single initial migration is fine for greenfield; split only if schema changes post-launch

## Security Considerations
- `encryptedKey` column for API keys (never store plaintext)
- `keyHash` for agent API keys (SHA-256, never reversible)
- `DATABASE_URL` only in `.env`, never in code
